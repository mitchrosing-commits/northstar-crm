# Northstar CRM Current Status

## Implemented MVP Capabilities

- Multi-tenant workspace data model with users and memberships.
- Current request/workspace context resolves a real active user, workspace, and membership before workspace-scoped page/API work proceeds.
- A narrow auth session abstraction supports local signed-cookie sessions, a configured trusted user-id header, and seeded demo fallback behind explicit auth modes.
- A minimal local auth MVP is available with email/password signup/login, clean first-workspace creation for new signups, signed httpOnly session cookies, server-side session lookup, expired-session rejection, login-time expired-session cleanup, and logout/session revocation.
- Password Reset MVP is available for existing local-login users with hashed reset tokens, expiry, one-time consumption, generic request responses, queued Resend or webhook reset-link email delivery, and dev/test-only reset link display for manual QA.
- Account Settings MVP is available from Settings for signed-in users to view their account name/email and make display-name updates only.
- The app shell shows the current signed-in user's name or email.
- The app shell uses a focused primary navigation for Dashboard, Assistant, Inbox, Pipeline, Deals, Quotes, Contacts, Organizations, Leads, Activities, Meeting Intelligence, and Settings. A compact command area keeps workspace search and jump shortcuts available, while basic create actions stay on their relevant list/module pages instead of dominating the sidebar.
- Workspace role helpers centralize `OWNER`, `ADMIN`, and `MEMBER` labels plus simple owner/admin/member checks.
- Authenticated users with multiple memberships can switch the active workspace from the app shell; the selection is persisted in an httpOnly cookie and validated against current memberships.
- Authenticated users can create a new workspace from Settings, become its owner, and have it selected as the active workspace immediately after creation.
- Workspace Settings shows the current workspace, current user's role, settings-access status, account display-name settings, active member list, conservative role editing and ownership-transfer controls, member-removal controls, a simple workspace creation form, and owner/admin invitation controls with shared admin panel spacing plus responsive member and invitation tables.
- Settings includes a Developer/API overview for the workspace-scoped REST surface, OpenAPI and route-map references, CSV export endpoints, and planned disabled API-key/webhook controls.
- Workspace invitations are email-based: owners/admins can create and revoke pending invitations, configured deployments automatically queue invitation email through the same Resend/webhook background-worker email path used for auth email, invitees can sign up or sign in with the invited email, accept the shared link, and the accepted workspace becomes active immediately. Manual accept links remain visible in Settings when delivery is missing or delayed.
- Owners/admins can remove non-admin workspace members from Settings, while only owners can remove admins. Workspace owners can promote/demote Admin/Member roles and transfer ownership to another active member, with audit logs for role changes, ownership transfer, and member removal.
- Role and account permission boundaries are pinned by source and integration tests covering workspace member management, ownership transfer, cross-workspace membership targeting, and display-name-only account updates.
- Runtime environment validation checks required database configuration and optional deployment/demo values.
- A non-sensitive `/api/health` readiness route validates runtime env and database connectivity.
- Local/dev migration readiness is documented: after pulling committed Prisma migrations, run `npm run prisma:deploy` before starting the app or browser smoke tests.
- Development actor/workspace fallback still uses seeded users and the demo workspace when `AUTH_MODE=demo`; trusted user-id headers are honored only in `AUTH_MODE=trusted-header`.
- Workspace API boundaries consistently return `401` for missing sessions, `403` for non-members, and non-leaky `404` responses for missing or cross-workspace records where appropriate.
- Workspace API singleton and nested action routes reject unsupported child path segments with the same stable not-found response instead of treating malformed nested paths as valid record or mutation routes.
- A workspace-scoped CSV export is available for Deals, Contacts/People, Organizations, Leads, Activities, Products, and Quotes from Settings. Settings export cards show workspace row counts, custom-field column availability where supported, and header-only behavior when no rows exist. Pipeline can jump directly to the Deals CSV export, and Deal, Contact, Organization, Lead, and Activity list pages can export the current filtered/search/sort view; Deals export includes compact commercial columns for line-item count, quote count, latest quote status/number/total, and CRM record exports include custom fields for the exported record type where supported.
- Settings Import / Export includes explicit import safety rules, sample CSV templates, horizontally scrollable row-preview tables for narrow screens, shared ready-to-import copy after valid previews, and row-level reporting for rare post-preview creation failures.
- Organizations CSV import is available from Settings using pasted CSV text, with required-name validation, unsupported-column reporting, workspace-scoped owner-email validation, duplicate detection, conservative creation of valid non-duplicate organizations, and `organization.imported` audit logs.
- Contacts CSV import is available from Settings using pasted CSV text, with required-name validation, case-insensitive email duplicate detection, unsupported-column reporting, workspace-scoped owner-email and organization-name reference checks, conservative creation of valid non-duplicate contacts, and `contact.imported` audit logs.
- Leads CSV import is available from Settings using pasted CSV text, with required-title validation, status validation, unsupported-column reporting, workspace-scoped owner-email, contact, and organization reference checks, duplicate-title detection, conservative creation of valid non-duplicate leads, and `lead.imported` audit logs.
- Pipeline board grouped by stages.
- Deal create/edit, detail view, stage movement, won/lost close and reopen flow with persisted `wonAt`/`lostAt` outcome timestamps, notes, activities, and audit history.
- Product Catalog is available from `/products` with workspace-scoped product creation, editing, deactivation/reactivation, listing, and active product pricing.
- Open deal detail pages can add/remove product line items, preserve product name/price snapshots, and show a separate line item total with responsive item tables. Closed deals reject line item additions/removals through the normal deal edit lock.
- Deal detail pages can create, review, browser-print, download on-demand PDFs, and internally track Quote statuses from current deal line items; quote draft and quote detail items snapshot same-workspace line item names, descriptions, quantities, prices, currency, and totals with responsive item tables.
- Quote-to-Cash / Commercial Workflow v1 adds a shared commercial readiness helper plus deal detail, quote detail, Deals list, pipeline card, dashboard, saved-view/filter, and export signals for line items, quote count/latest quote, accepted quote/deal value alignment, customer context, and contract/SOW state.
- Draft quotes support one quote-level discount and one quote-level tax adjustment. Percent values are stored as basis points, fixed values are stored as cents, subtotal remains the item snapshot total, percent tax is calculated after discount, final totals cannot go below zero, and sent/accepted/declined quotes lock adjustment editing.
- Authenticated workspace members can generate and revoke secure public quote links. Public quote pages render quote snapshot details at `/q/:token` without the CRM app shell, internal navigation, audit data, workspace member data, or rendered internal IDs, are marked noindex, and allow customer acceptance only from an active, unexpired link while the quote is `SENT`.
- Accepted quote snapshot totals can be manually synced into the linked open deal value and currency from quote detail. Sync writes an audit log, is a no-op when the deal already matches the accepted quote, and respects closed-deal edit locking.
- Open deal detail pages include a database-backed contract workflow for the NDA -> MSA -> SOW sales agreement path. Each workspace-scoped deal contract step can track status, owner, due/sent/signed dates, notes, and an external document reference placeholder, with audit logs for creation and updates. Contract step dates are validated before writes. Closed deals display contract workflow steps read-only and reject contract-step mutations.
- Deals list and pipeline cards surface compact contract workflow summaries from real deal contract steps, falling back to legacy NDA/MSA/SOW custom field values when no step rows exist yet. Deals list also supports commercial filters for open deals without quotes, deals with quotes, accepted quotes, and open deal value without line items.
- Lead list/detail/create/edit and lead-to-deal conversion, with a summary-level jump to the conversion panel on Lead detail pages.
- Contact list/detail/create/edit with linked deals, activities, and notes.
- Organization list/detail/create/edit with linked people, deals, activities, and notes.
- Deal, Contact, Organization, Lead, and Activity forms share owner-assignment guidance for unassigned records and single-member workspaces.
- Activity creation from deals, contacts, organizations, and leads.
- Activity completion plus open-activity edit/reschedule/delete; due/completed timestamps are validated before writes, and completed activities are locked from normal edits and deletion.
- Activity edit pages show a shared workspace summary for status, due timing, type, owner, and linked CRM record context before edit or locked-completed states.
- Global Activity creation page can create a follow-up for an existing open deal, contact, organization, or unconverted lead. Prefilled follow-up links can carry a safe internal return destination so save/cancel returns users to the originating list, dashboard, record, quote, or email surface.
- Manual email logs can be created from Deal, Contact, Organization, and unconverted Lead detail pages. Logged emails are workspace-scoped snapshots, appear in record timelines, and do not send email or run whole-mailbox sync.
- Workspace email templates can be created, edited, deactivated, and reused to prefill manual email logs. Deactivated templates remain manageable in Settings but are not offered in record log forms.
- AI Email Reply Assistant v1 is available from stored email cards in Inbox when `OPENAI_API_KEY` is configured. It assembles workspace-scoped email, contact, organization, deal/lead, recent note/activity, quote/contract/product, Meeting Intelligence, and curated Relationship Brief context, then returns an editable subject/body draft with context-used metadata and review cautions. Meeting Intelligence meeting summaries and approved follow-up activities can appear as CRM context for a linked email thread or log, while Relationship Brief facts are passed with deterministic field-level usage guidance and internal-only guidance is withheld from customer-facing draft context. Drafts are review-first only: Northstar does not send them automatically.
- Smart Email Labels v1 are available from stored email cards in Inbox. Stored email cards show deterministic local label suggestions from the same local rules used by Work Inbox tags, even when AI is unavailable, and users with `OPENAI_API_KEY` configured can explicitly refine those labels with AI. If the provider is missing or fails, Northstar saves or keeps local-rule labels with a sanitized notice instead of surfacing provider failure as the primary UI state. Explicit classification saves suggested category/signal labels, confidence, flat fallback evidence, optional category-level evidence, optional per-signal evidence, and cautions on the email log, and users can review saved urgent, needs-reply, pricing/quote, contract/legal, relationship-risk, and buying-signal items in the Relationship Inbox Queue. Labels are suggestions and do not send email, mutate Gmail labels, or create activities, notes, leads, CRM links, or profile facts. Older Smart Label snapshots that only contain flat evidence still read and render through the queue evidence fallback.
- Relationship Inbox follow-up drafting is available from stored email cards. Saved Smart Labels, email metadata, evidence, and linked CRM context prefill an editable activity draft for the linked deal, lead, contact, or organization; users must review and submit before `createActivity` writes the follow-up through the normal activity service, validation, audit, and timeline path. Reviewed email follow-ups also create an `EmailLogActivityLink` provenance row so the source email and created activity stay associated even if the activity description is edited later, and stored email cards show linked follow-up activity history with open/completed status, direct activity review links, and inline completion for open linked follow-ups through the existing activity completion path.
- Relationship Inbox Queue on `/email` filters stored emails by priority/signal/category, including urgent, needs reply, follow-up needed, relationship risk, pricing/quote, contract/legal, waiting on customer, potential leads, customers, and prospects. Unclassified inbound or CRM-linked emails can appear with a classify-first recommendation instead of being hidden. Queue rows show subject, sender/date, linked CRM record, top labels, confidence when saved, durable follow-up state, compact linked follow-up counts, a compact "why this?" explainer with deterministic evidence from saved categories/signals, saved evidence, CRM link state, and durable/legacy follow-up state, plus a deterministic next-best-action summary with label, reason, severity, signal-to-action explanation, and primary review/open/complete action to classify, link a CRM record, draft a reply, review relationship risk or potential-lead context, review/create follow-up work, complete the exact open linked follow-up activity, or acknowledge that no action is currently needed. Queue rows can jump to a full evidence section on the stored email card; that section shows category evidence, expandable per-signal drilldowns with saved reasons/excerpts when the snapshot has them, flat fallback excerpts for older labels, a "Why this action?" chain from saved signals/category to CRM/follow-up state to the recommended action, CRM/follow-up source metadata, and a clear note that exact source text offsets are not stored.
- Northstar Assistant diagnostics foundation v1 is available as a reusable AI operating-layer service plus a small UI panel on Inbox, Contact detail, and Deal detail pages. The service builds sanitized, workspace-scoped context for contacts, deals, leads, organizations, inbox/email, Meeting Intelligence proposals, and job/system diagnostics; deterministic detectors explain missing CRM links, stale or overdue follow-ups, misplaced contact/company Relationship Memory signals, unlinked priority email, missing durable follow-ups, connection readiness issues, and failed/stale jobs. The UI shows what was inspected, findings, evidence, confidence, guardrails, and review-first next actions. Viewing the panel does not mutate CRM records, retry sync, reconnect providers, classify email, send email, or apply Meeting Intelligence proposals. Optional provider summarization is gated behind `OPENAI_API_KEY` and receives only sanitized assistant context. The `/assistant` left-nav page offers deterministic command handling for today's agenda, highest-risk open deals, and stored-email reply checks using bounded workspace context. It also drafts review-first action previews for activities, contact Relationship Memory/profile updates, organization/contact creation, and AI preference changes. Draft cards show proposed fields, evidence, confidence, candidate matches, missing information, warnings, disabled apply wording, and a Save to review queue action. Saved drafts become sanitized, capped `AssistantActionRequest` rows with `PENDING` status and current-user/workspace scope; users can reject pending requests, which moves them to `REJECTED` and writes an audit event. `APPLIED` is reserved in the model for a future explicitly confirmed apply workflow, but no apply path exists yet. Saving or rejecting Assistant requests does not save settings, create/update/link/delete/convert/close CRM records, call providers, sync/send email, or mutate provider mail.
- Meeting Intelligence / File-to-CRM Intake is available at `/meeting-intelligence` with persistent `MeetingIntake` records. Users can paste notes or upload local text/markdown/RTF/HTML/CSV/JSON/PDF/DOCX files, add context and optional CRM record hints, review normalized markdown with source/extraction metadata, grouped matched CRM records, unmatched mentions, evidence/confidence metadata, proposed completed meeting activity, proposed notes, proposed Relationship Brief updates, and proposed follow-up activities, edit/select the proposal, manually reassign known CRM targets, clear uncertain targets, and apply only approved updates. Relationship Brief suggestions are deterministic by default; when `MEETING_INTELLIGENCE_RELATIONSHIP_PROVIDER=openai` and `OPENAI_API_KEY` are configured, a provider-backed semantic extractor can enrich matched-contact proposals with provider attribution, sensitivity guidance, warnings, and merged previews while keeping deterministic fallback. Relationship Brief review supports fact-level include/edit/reject controls, moving facts between profile fields, duplicate detection against existing brief text, and staleness warnings for time-sensitive facts. Retargeting a Relationship Brief proposal to another contact reloads that contact's current Relationship Brief through the workspace-scoped contact API, refreshes Existing and After apply previews, recalculates duplicate/staleness guidance, and blocks apply while the target preview is unavailable. Apply validates submitted targets in the current workspace and skips unavailable, closed-deal, converted-lead, deleted, or cross-workspace targets with clear reasons. Applied updates create normal notes and activities through existing services, can merge approved Relationship Brief updates into matched contact profiles, preserves audit logs, workspace scoping, closed-deal locks, converted-lead locks, and activity completion rules. Meeting Intelligence-created follow-up activities carry an explicit source line in their description, appear in normal activity lists/timelines, can be completed through the normal activity completion path, and do not create `EmailLogActivityLink` rows unless the follow-up originated from an email review. Relationship Brief apply results now include a compact field-level change summary, and contact detail Relationship Brief panels show recent audit-backed changes with previous/new excerpts, Meeting Intelligence source context, actor, timestamp, and accepted-fact counts. Reopening an already-applied intake returns the stored apply result and does not duplicate CRM updates.
- Meeting Intelligence routes noteworthy facts to the current safest destination instead of forcing every memory into a contact profile. Contact/person facts can be proposed as curated Relationship Brief updates when they match existing profile fields and as contact timeline notes when raw memory is more appropriate. Organization/company facts are proposed as organization notes, deal-specific facts as deal notes or meeting context, and lead-specific facts as lead notes when the lead is still unconverted. Review labels distinguish curated Relationship Brief changes, raw contact notes, company notes, deal notes, lead notes, meeting logs, and follow-up activities before apply.
- Meeting Intelligence source detection recognizes pasted text, markdown, text files, RTF, HTML/HTM, CSV, JSON, PDF, DOCX, PPTX, XLSX, image/whiteboard, audio, video, and unsupported artifacts. Text, markdown, RTF, HTML, CSV, JSON, text-based PDF, and DOCX extraction are implemented locally with bounded inputs and no original-file storage. Images, scanned/image-only PDFs after local text extraction returns empty, audio, and provider-capable video are stored in private Meeting Intelligence file storage and queue `meeting_intake.extract_media` jobs with a storage reference, not the original base64 payload, when `MEETING_INTELLIGENCE_MEDIA_PROVIDER_URL` is configured. Storage defaults to local filesystem for development/single-instance use and can be switched to S3/R2-compatible object storage for production/multi-instance deployments. The S3/R2 backend supports single-object direct upload and durable-resumable multipart upload for provider-backed files: the browser computes SHA-256, creates a workspace-scoped upload session, uploads bytes directly to private object storage, then finalizes or completes the draft intake so extraction is queued from `storedFile` without file bytes entering the Next.js intake request body. Multipart resume stores one safe local record per workspace with session id, file label, MIME type, byte length, checksum, part policy, uploaded ETags, and expiry, then inspects the server session and reconciles S3/R2 ListParts after reload before continuing from the next missing part. The upload capabilities endpoint advertises safe environment policy before bytes move, including direct-upload availability, multipart availability and part policy, local/provider/unsupported source support, max sizes, cleanup posture, and retention; the form uses it to show direct/multipart/bounded upload guidance and block unsupported or too-large files before hashing/uploading. The form shows hashing/session/upload/finalize/complete/queued/fallback/resume/cancel/aborting/failed states, retries single-object signed PUT once only while the URL is still safely valid, signs multipart parts one at a time, and preserves multipart state after transient upload failure so the user can resume or explicitly cancel. Session metadata is cleanup-visible before content arrives; finalization/completion requires the same size/checksum/workspace/intake/source/expiry validation used by the worker; repeated finalization/completion cannot create duplicate extraction jobs. Local/dev, small local-extraction files, unsupported upload source types, missing provider config, and safe upload-session failures keep the existing bounded base64 fallback. The worker retrieves and verifies the stored bytes before provider extraction, deletes the stored file on successful extraction, and leaves retryable failures or abandoned incomplete multipart metadata available until retention cleanup. When the provider is missing, unsupported, or fails, the intake persists clear provider status and no fake extracted content. Northstar now includes `/api/internal/meeting-intelligence/media-extract` as a provider-compatible internal route: with `MEETING_INTELLIGENCE_MEDIA_PROVIDER=openai` and `OPENAI_API_KEY`, it handles image/whiteboard OCR/vision and audio transcription. Scanned/image-only PDFs and video remain explicitly unsupported by the internal OpenAI route until safe PDF OCR and video audio-extraction paths or a provider capable of those source types are added. PPTX and XLSX remain unsupported pending a direct local parser or export to PDF/DOCX/CSV/HTML/markdown/text, and legacy `.doc` remains unsupported.

- The Developer/API settings surface documents available workspace REST areas for Deals, Contacts/People, Organizations, Leads, Activities, Notes/timeline inputs, Custom Fields, Email Templates, Quotes, Contract Workflow, Products/line items, Import/Export, and internal job commands.

- Consistent audit/history display for deals, leads, contacts, organizations, and dashboard recent changes.

## Known Limitations

- Full account management is not implemented. Account Settings currently supports display-name updates only; email change, avatar upload, password change, account deletion, SSO/OAuth, and 2FA are not implemented.
- The auth/session helper supports local signup/login, local password reset for existing users, trusted-header auth, and demo fallback. SSO, OAuth providers, 2FA, and billing are not implemented.
- Password reset email delivery supports direct Resend delivery or the provider-neutral webhook fallback, and is queued through the `Job` table. Production reset requests enqueue `auth.password_reset_email` jobs only when an absolute reset URL can be built from a public HTTPS `APP_BASE_URL`; `npm run jobs:work` can process queued reset email jobs continuously, and `npm run jobs:run-once` remains available for manual or scheduled one-batch processing. Repeated reset requests can enqueue multiple email jobs, but each new request consumes prior active reset tokens so stale queued links fail safely, and expired or no-longer-current reset-email jobs are marked complete without sending dead links. Missing config is reported as a readiness warning, and missing or failed delivery still returns the same generic response and never displays reset links.
- Workspace invitation email delivery uses the same `APP_BASE_URL` plus Resend/webhook auth email configuration and the `workspace.invitation_email` job. Missing email config does not block invitation creation; manual accept links remain visible in Settings. Revoked, accepted, or deleted-workspace invitations are skipped by the worker instead of sending stale links.
- Trusted-header mode must run behind a trusted reverse proxy or auth gateway that strips client-supplied auth headers before setting the configured user-id header.
- Local auth mode requires `AUTH_MODE=local` and `AUTH_SESSION_SECRET`; existing seeded users with password hashes can sign in, and new users can sign up into a clean workspace. It is intentionally not a full account-management product.
- Workspace roles are visible and reusable in code, but advanced permissions, visibility groups, and row-level permissions are not implemented.
- Workspace switching is limited to existing memberships plus workspaces the signed-in user creates or accepts by invitation. Broader member management, ownership transfer beyond a single current owner handing off to another active member, and workspace deletion are not implemented.
- Workspace display names are required, trimmed, capped at 120 characters, and may be duplicated; new workspaces receive unique generated slugs for routing/API identity.
- Invitees without an account can create one with the invited email before accepting, and `OWNER` invitations are blocked.
- Invitation management is intentionally narrow: duplicate pending invitations are rejected, pending invitations can be revoked, accepted invitations are idempotent only while the accepted membership still exists, removed members cannot rejoin from old accepted links, and invitations still cannot grant `OWNER`.
- Member management is intentionally narrow: normal members cannot remove others or edit roles, Admins can manage settings but cannot promote/demote admins, remove admins, or transfer ownership, normal role edits cannot assign `OWNER`, current-owner transfer demotes the previous owner to `ADMIN`, current owner removal remains blocked, and the service blocks changes that would leave the workspace without an owner/admin.
- Custom fields are currently supported for Deals, Contacts/People, Organizations, and Leads in the UI.
- Editable custom field types are `TEXT`, `NUMBER`, `DATE`, `BOOLEAN`, and `SELECT`.
- Seed data includes realistic editable custom field definitions and values across Deals, Contacts/People, Organizations, and Leads.
- Unsupported future custom field types display as read-only with plain-language guidance until explicit inputs and validation are implemented.
- Closed deals and converted leads display custom fields read-only and reject custom field value updates.
- Record create forms keep base-record creation atomic; custom field values are filled after creation from detail or edit pages.
- Core list search is limited to simple case-insensitive matching on visible CRM fields; it is not full-text search.
- Custom field list filtering is limited to one supported custom field at a time. The MVP supports `equals`, text-only case-insensitive `contains`, `is_empty`, and `is_not_empty`; number/date comparisons, select/multi-select filtering, multiple filters, richer saved views, reporting, searching, reordering, and permissions are not implemented.
- Saved views are currently available for Deals, Leads, Contacts, and Organizations, workspace-wide, and limited to saving/reapplying/deleting list state. Filtered/saved list headers are visible on supported list pages, and the core saved-view list tables use responsive horizontal scroll regions. Activities would require an intentional saved-view record-type/schema slice, Products have no comparable filter surface, and sharing, ownership, permissions, and broader cross-record saved views are not implemented.
- Deal Reporting v1 is a simple workspace-scoped pipeline insight page; it can respect Deals list query parameters and includes a lightweight Forecasting v1 section, but it does not create saved reports.
- Product Catalog and Deal Line Items are MVP-only. Product CSV export is implemented, but product import, price books, inventory, line-item discounts, tax rules/jurisdictions, and subscriptions are not implemented.
- Quote lifecycle supports internal tracking plus customer acceptance from an active, unexpired public link while a quote is `SENT`: `DRAFT -> SENT -> ACCEPTED` or `DRAFT -> SENT -> DECLINED`. Browser print views and on-demand PDF downloads are available for authenticated users, and public quote links can be generated only for sent quotes through secure tokens with responsive, customer-facing quote item tables. Internal and public quote reads suppress quote items and direct deal contact/account labels when persisted relations are inconsistent across workspaces. Draft, revoked, expired, missing, or malformed public quote tokens return a safe 404. Quote subtotals, fixed adjustment values, calculated discount/tax amounts, and totals are validated against current integer storage limits. PDF file storage, e-signature, email sending, payment collection, line-item discounts, tax rules, and tax jurisdictions are not implemented.
- Quote-to-deal value sync is explicit, accepted-quote-only, and limited to open deals. Deal reports and exports reflect synced quote totals only because the deal value changes after a user runs the manual sync action.
- Line item totals do not overwrite deal value or reporting totals. Deal reports and exports continue to use the existing deal value field. Line items remain mutable only while the deal is open. Product prices, quantities, and computed line-item totals are validated against current integer storage limits.
- Contract workflow is local CRM tracking only. OpenContracts templates, document generation/storage, redlining, approvals, and e-signature are future integration layers and are not implemented in this slice.
- Dashboard pipeline health is a lightweight operating summary based on open deals, overdue/due-today activities, open deals without a next activity, and active leads without a next activity; no scoring model, goals, forecast history, saved reports, or report builder is implemented.
- Forecasting v1 is open-deal-only and current-state-only. It groups currencies separately with no FX conversion, weights only deals whose current stage has a probability set, and does not use closed-deal timing, lost reasons, forecast history, snapshots, goals, quotas, forecast categories, or direct line-item forecast inputs.
- Deal close outcome timestamps are forward-looking. Existing deals closed before the `wonAt`/`lostAt` fields were introduced can remain null because there is no safe historical close-date backfill source.
- Goals v1 is intentionally narrow. It supports workspace-level monthly won-revenue targets using same-currency `WON` deals with `wonAt` inside the month, validates targets against current integer-cent storage limits, excludes legacy won deals with null `wonAt`, and does not use `createdAt`, `updatedAt`, or `expectedCloseAt` as fake close dates. Admins and owners can save workspace goals; members can view goal progress. No charts, dashboard widgets, owner/user/team goals, quarterly goals, activity goals, or FX conversion is implemented.
- No charting, forecast history, saved reports, scheduled reports, or report builder is implemented.
- Activities are manual CRM follow-ups only; recurring tasks, notifications, reminders, calendar/email sync, and broad automation are not implemented. Due-date shortcuts set dates only; they do not create reminders. Deal-attached activities on closed deals and lead-attached activities on converted leads are read-only for update, completion, and deletion. One-click deal activity templates are lifecycle-aware: open-deal templates reject closed deals, while won/lost-specific follow-up templates remain available for their matching closed outcome.
- Activity saved views are not implemented. Activities would require an intentional saved-view record-type/schema slice, and Products still have no comparable saved-view filter surface.
- CSV export is intentionally limited to Deals, Contacts/People, Organizations, Leads, Activities, Products, and Quotes; notes, saved views, users, memberships, auth/session data, and infrastructure configuration are not exported.
- CSV related-record labels, deal commercial counts/latest quote columns, and custom field columns are scoped to the exported record type and workspace. CRM list/detail reads and relation-backed search also suppress direct Deal, Lead, Contact, and Activity labels when persisted relation IDs are inconsistent across workspaces. Duplicate custom field display names are disambiguated with the field key in the export header. Simple stored values are exported; complex unsupported custom field JSON values are left blank. Export formatting quotes delimiter/newline cells and prefixes spreadsheet formula-looking values with an apostrophe. List-page exports for Deals, Contacts/People, Organizations, Leads, and Activities respect the current search/filter/sort query, including existing CRM record custom-field filters where supported, but export all matching rows rather than just the current pagination page.
- CSV import accepts the documented camelCase headers plus human-readable spaced aliases for supported relationship/date fields, such as `Owner Email`, `Contact Email`, `Organization Name`, `Pipeline Name`, `Stage Name`, and `Expected Close At`.
- Organizations import creation is conservative: it creates valid non-duplicate organizations after server-side revalidation, skips duplicates and invalid rows, and never overwrites or updates existing organizations.
- Organization import preview supports `name` plus optional `domain` and `ownerEmail`. Custom field columns and other export-only columns are reported as unsupported and ignored, while skipped rows receive stable skip reasons.
- Contacts import creation is conservative: it creates valid non-duplicate contacts after server-side revalidation, skips duplicate emails and invalid rows, and never overwrites, updates, or merges existing contacts.
- Contacts import supports `name` or `firstName`, plus optional `lastName`, `email`, `phone`, `organizationName`, and `ownerEmail`. Organization and owner references must already exist in the current workspace and ambiguous organization names are reported as invalid.
- Leads import creation is conservative: it creates valid non-duplicate leads after server-side revalidation, skips duplicate titles and invalid rows, and never updates, overwrites, merges, or converts leads.
- Leads import supports `title` or `name`, plus optional `source`, `status`, `contactEmail`, `contactName`, `organizationName`, and `ownerEmail`. Blank status imports as `NEW`; `NEW`, `QUALIFIED`, and `DISQUALIFIED` are accepted. `CONVERTED` status and conversion behavior are intentionally excluded. Contact, organization, and owner references must already exist in the current workspace and ambiguous names are reported as invalid.
- Deals import creation is conservative: it creates valid non-duplicate deals after server-side revalidation, skips duplicates and invalid rows, never updates, overwrites, or upserts existing deals, and writes `deal.imported` audit logs.
- Deals import supports `title` or `name`, required `pipeline` or `pipelineName`, required `stage` or `stageName`, plus optional `status`, `value`, `currency`, `expectedCloseAt`, `contactEmail`, `contactName`, `organizationName`, and `ownerEmail`. Deal values must fit the current integer-cent storage limit. Pipeline/stage, owner, contact, and organization references must already exist in the current workspace; cross-workspace or ambiguous references are invalid.
- Deals import can create `WON` or `LOST` rows by status, but it does not infer `wonAt`/`lostAt` or lost reasons. Imported won deals with null `wonAt` are excluded from Goals v1 progress unless later reopened and closed in-app.
- CSV import creation does not support Activities, Notes, Products, Quotes, saved views, users, memberships, or custom fields. Deal custom field import remains deferred; `Custom: ...` columns are reported as unsupported and ignored.
- Manual email logging is plain-text/metadata only, and new manual email logs are limited to open deals, unconverted leads, contacts, and organizations. The `/email` page is now treated as the Inbox module surface: Full Inbox is the synced Gmail mailbox/thread reader and primary connected view, while Relationship Inbox remains the advanced prioritized CRM action queue for stored/classified email. Full Inbox opens as an All-tab list with no selected message by default, defaults to newest received first, and offers explicit URL-driven filters/sorts including Hide unimportant, unread-first, oldest-first, and priority; selected threads replace the list with a focused reader, and Back to inbox preserves the current account, tab, search/filter/sort, page, and page-size URL state. Smart Labels, AI Reply Assistant drafting, and email follow-up creation/history live inside this Inbox workflow rather than becoming separate left-nav modules. Gmail Full Inbox v1 uses the background job worker for safe incremental inbox sync: connecting/reconnecting Gmail enqueues an `email.gmail_sync` job, the worker prefers Gmail history cursors after an initial recent inbox sync, and expired/missing history falls back to a bounded recent inbox batch. Clicking `/email` Sync Gmail inbox enqueues and immediately claims one bounded Gmail sync job through the same job record, so explicit user retries/testing can move beyond queued even when the continuous worker is missing; stale pending jobs are claimed, stale retryable running jobs are recovered before claim, retry-pending jobs with a previous failure show a sanitized retry/failure status instead of plain queued copy, and retry-exhausted stale running jobs are dead-lettered with a safe user-facing failure instead of trapping the account. The current Gmail import bounds are 75 recent messages by default and 100 messages for explicit job-backed sync/load-more paths. Unattended reconnect/background jobs still require `npm run jobs:work` or scheduled `npm run jobs:run-once`. The Gmail sync progress panel is now available from collapsed advanced diagnostics, while the main inbox shows a subtler sync status/action strip. The `/email` Load older messages action uses a user-triggered bounded Gmail `before:` inbox search based on the oldest stored thread message, and the selected-thread Refresh action fetches Gmail thread detail for a thread already present in the current workspace; neither path mutates the stored history cursor. The page distinguishes setup-required, token-encryption-required, reconnect-required, connected, sync-issue, first-sync, and synced-empty states with user-facing status copy. Manual logging remains available as a legacy fallback for messages not yet available through Full Inbox and should be removed or further de-emphasized after Gmail Full Inbox is proven in boss testing. Synced messages are stored as deduped `EmailLog` snapshots and explicit Gmail replies remain user-submitted from `/email`; Gmail sync and inbox triage do not auto-send, auto-classify with a provider, auto-create CRM records, delete/archive/mark-read provider mail, sync attachments, preserve full HTML rendering, track opens/clicks, or handle unsubscribes. Microsoft/Outlook whole-mailbox or background sync, general SMTP sending, inbound email processing beyond Gmail snapshots, Smart Bcc capture, CRM email automations, and API keys are not implemented.
- AI email reply generation is draft-only and provider-optional. Missing `OPENAI_API_KEY` disables generation without breaking manual email logging, templates, or sync review. Generated drafts are not persisted, not sent automatically, and must be reviewed for pricing, discounts, legal terms, dates, and promises before use.
- Smart Email Labels are explicit, provider-optional, and saved only as classification snapshots on stored email logs. Missing `OPENAI_API_KEY` still allows deterministic local-rule labels, while provider-backed AI refinement remains disabled until configured. Labels do not trigger workflow automation, lead creation, activity creation, note creation, Relationship Brief updates, sending, syncing, Gmail label mutation, or external inbox changes.
- Email-to-follow-up activity drafting does not require AI provider config once a stored email exists. Draft defaults can use saved labels when available or conservative manual defaults when labels are missing. Unlinked stored emails show a link-first message because normal activities must attach to a CRM record.
- Relationship Inbox Queue follow-up state uses `EmailLogActivityLink` first, so reviewed follow-ups remain tied to their source email after activity description edits and can show created vs. completed from the linked activity status. The queue and source email cards surface the linked activity title, due/completed timing, related CRM record, and `/activities/:id/edit?returnTo=/email` review destination. Open linked follow-ups can be marked complete inline from `/email`, which refreshes the queue state through normal activity completion semantics without creating duplicate activities. Multiple linked follow-ups are ordered with open work first, then due-date order, and the queue stays in follow-up-created state while any linked follow-up remains open. Deterministic next-best-action and "why this?" rules prefer durable linked follow-ups over legacy marker matches, never create CRM records automatically, and use saved labels, saved category/signal evidence, flat fallback excerpts, CRM linkage, severity, and linked follow-up completion state only to explain the suggested review action. The expanded evidence trail keeps those reasons inspectable with source metadata for Smart Labels, CRM link state, durable follow-up links, legacy marker fallback, next-best-action reasoning, and service-layer "Why this action?" copy that connects contributing signals to reply, follow-up, risk-review, lead-review, completion, or no-duplicate-work recommendations. When the primary recommendation is to finish existing work, the queue targets the exact linked open activity and reuses the normal activity completion control. Older activities without that durable link can still be recognized by the legacy `Source email: ...` description marker when the activity is attached to the same CRM record; if neither durable nor legacy evidence exists, state remains no follow-up or unknown rather than inferred from unrelated work.
- Email provider settings show configuration readiness, Gmail / Google Workspace and Microsoft 365 / Outlook OAuth connect paths, background sync status, and sync actions only after a provider is connected with sufficient scopes. Provider OAuth requires env vars plus `EMAIL_TOKEN_ENCRYPTION_KEY`; IMAP / SMTP is still planned/disabled. Gmail Full Inbox requires Gmail read/send scopes and asks older metadata-only connections to reconnect before inbox sync or replies. Gmail sync stores full readable bodies when available, snippets, labels, provider ids, and unlinked messages, dedupes by provider message id, stores the Gmail history cursor in `EmailConnection.lastSyncCursor`, and never stores OAuth tokens or message bodies in job payloads. Manual load-more and thread-refresh actions update `lastSyncAt`/`lastError` only, leaving the background history cursor owned by the worker path. `/email` can disconnect active Gmail or Microsoft provider tokens without deleting stored email logs. Microsoft sync still reads recent message metadata/snippets, matches known contacts, dedupes by provider message id, constrains synced log attachments to same-workspace records, and skips unmatched messages, full bodies, attachments, labels, deletes, sends, and whole-mailbox import.
- API keys, webhook subscriptions, OAuth app installs, and external developer portals are not implemented. The Developer/API settings page is an honest preview of the current REST surface and planned platform controls.
- Background jobs runtime is limited to the internal `Job` table/service foundation, safe aggregate queue-status inspection, a single-run worker shell, a continuous worker mode with stale `RUNNING` recovery and max-attempt dead-lettering, aggregate-only terminal cleanup plus Meeting Intelligence stored-file retention cleanup, `internal.noop`, queued password-reset email delivery, queued workspace-invitation email delivery, configured Meeting Intelligence media extraction, and queued Gmail Full Inbox sync. No broader event outbox, automation, reminder, webhook platform, integration, or async import runtime is implemented yet.
- Fast tests are mostly source-level checks; integration tests require a guarded `TEST_DATABASE_URL`.
- Deployment/readiness documentation exists with a minimal Railway preview configuration; other hosting-provider infrastructure is not implemented.
- Mobile responsiveness has basic support and automated browser smoke coverage for a narrow viewport subset.
- OpenAPI is hand-maintained and should eventually be generated or contract-tested.
- Service code has been split into focused domain modules behind the `lib/services/crm.ts` compatibility barrel.
- Unified timelines are read-only and lightweight; they combine notes, activities, manual email logs, and audit events. Calendar history, attachments, comments, and restore/undelete are intentionally deferred.
- Deal detail history uses existing notes, activities, manual email logs, and audit events only; external email/calendar history and generated summaries are not implemented.
- Account display-name updates and password reset requests are auth/account events outside workspace CRM audit history in this MVP.
- Notes and manual email logs are plain text only; rich text, attachments, mentions, comments, and AI summaries are intentionally deferred.
- Relationship Brief fields can be manually maintained on contacts with sensitivity guidance, and Meeting Intelligence can propose review-first Relationship Brief updates for matched contacts when meeting notes contain explicit personal context, communication style, business concerns, follow-up reminders, or internal guidance. A shared usage-guidance model defines labels, badges, customer-facing use, and AI/email usage for each field; personal context may inform light voluntary personalization, communication style informs tone, business concerns are cautious sales context, follow-up reminders are internal next-step memory, and internal guidance is internal-only/do-not-mention. Manual saves and approved Meeting Intelligence changes both write concise field-level `person.updated` audit metadata so the contact panel can show recent previous/new excerpts, source/field filters, changed-field summaries, and expandable read-only provenance details without creating a separate version-control system. Users must approve and may edit Meeting Intelligence suggestions before any profile field changes, and can accept, reject, edit, or move individual proposed facts inside a Relationship Brief update. Duplicate facts already present in the contact profile default to excluded and are skipped during merge; date-like or event-like facts carry staleness warnings, and no AI/provider writes personal context automatically. Semantic extraction is opt-in and provider-backed only when `MEETING_INTELLIGENCE_RELATIONSHIP_PROVIDER=openai` and `OPENAI_API_KEY` are configured; provider failures fall back to deterministic suggestions.
- Meeting Intelligence is a CRM intake workflow, not a document-management, video, transcription, or AI provider product. It stores extracted text/markdown and review/apply JSON, plus temporary private stored-file metadata for provider-backed extraction; it does not expose or retain original binary files as user-facing CRM documents. Local text, RTF, HTML, CSV, JSON, text-based PDF, and DOCX processors run without external AI/OCR/transcription calls. Image OCR/vision, scanned PDF OCR, audio transcription, and provider-capable video extraction can run through the configured provider-neutral `MEETING_INTELLIGENCE_MEDIA_PROVIDER_URL` adapter and background worker; the first-party internal OpenAI route supports only image/whiteboard extraction and audio transcription when the required env vars are configured. Scanned PDFs and video remain unsupported by that internal OpenAI route. The implemented storage backends are private local filesystem storage and S3/R2-compatible object storage; S3/R2 supports direct and resumable multipart browser uploads, while local storage remains server-mediated. Multiple simultaneous resumable upload recovery, parallel multipart upload, and resume after browser storage is cleared remain unimplemented. Missing provider config, missing/expired stored files, checksum mismatches, PPTX/XLSX, unsupported internal-route media paths, and provider failures remain explicit boundaries surfaced as processor status metadata. The default deterministic v3 analysis provider returns reviewable proposals with evidence/confidence metadata; opt-in semantic Relationship Brief extraction can enrich matched-contact proposals with sensitivity guidance, but external AI providers must remain review-first. No external AI, OCR, transcription, or real object-storage call runs in tests.
