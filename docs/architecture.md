# Architecture

This CRM starts as a single Next.js TypeScript application with API route handlers, Prisma, and PostgreSQL. Backend code is organized around workspace-scoped service modules so it can later move into a dedicated API service without changing the public REST contract.

## Tenancy

Every major CRM record belongs to a `Workspace`. Current request context is resolved in `lib/auth/request-context.ts`, which uses the narrow session abstraction in `lib/auth/session.ts`. Local mode resolves a signed httpOnly session cookie to a non-expired, non-revoked server-side `Session` row and active `User`; trusted-header values are ignored in local mode unless a valid local session cookie is also present. Trusted-header mode maps a configured upstream user-id header to an active `User`. Seeded demo fallback is available only in demo mode. Workspace-scoped page/API work must resolve an active `WorkspaceMembership`. The resulting `{ workspaceId, actorUserId }` actor is passed into service functions. Services are still responsible for applying workspace filters, verifying membership with `ensureWorkspaceAccess`, and excluding soft-deleted records.

Workspace-scoped API boundaries use consistent access semantics: missing sessions return `401`, authenticated non-members return `403`, and missing or cross-workspace records return non-leaky `404` responses where resource existence should not be exposed.

Browser workspace selection is stored in an httpOnly `northstar_workspace` cookie containing only the selected workspace id. The cookie does not grant access: `getCurrentWorkspaceContext` accepts it only when the current user has an active membership in that workspace. Invalid, stale, or non-member selections fall back to the configured demo/default workspace when valid, then to the user's first active membership. API routes remain explicitly workspace-id scoped and do not use the browser active-workspace cookie.

Workspace creation is intentionally narrow. A signed-in user can create a workspace from Settings with a required trimmed display name. The creator receives an `OWNER` membership, the workspace creation is audit-logged, and the new workspace id is written to the same verified active-workspace cookie used by switching. Duplicate workspace names are allowed; the service generates a unique slug by suffixing when needed. Creation does not invite members, edit roles, transfer ownership, or create seed CRM data.

Workspace invitations are also narrow. `WorkspaceInvitation` records are workspace-scoped, store normalized invitee email, target role, status, inviter, and timestamps, and currently support `PENDING`, `ACCEPTED`, and `REVOKED`. Owners/admins can create or revoke pending invitations from Settings. Invites are for existing users only, do not grant `OWNER`, and do not send email; Settings exposes a manual accept link. A signed-in user can accept only an invitation whose email matches their account. Acceptance creates membership idempotently only while the accepted membership still exists, marks the invitation accepted, audit-logs the event, and selects the invited workspace through the same verified active-workspace cookie. If that member is later removed, the old accepted link cannot be reused to rejoin.

Workspace member management remains conservative. Owners/admins can remove normal members from Settings, owners can remove admins, and removal writes a `workspace_member.removed` audit log. Workspace owners can change another member between `ADMIN` and `MEMBER`, while normal role editing never assigns `OWNER`; those changes write `workspace_member.role_updated` audit logs. Ownership transfer is owner-only, targets another active membership, makes the target `OWNER`, and demotes the previous owner to `ADMIN` with a `workspace_member.ownership_transferred` audit log. The service blocks normal members from removing others, blocks admins from removing other admins, blocks non-owners from ownership transfer, keeps current owner removal blocked, and refuses changes that would leave the workspace without an owner/admin. Removing a member only removes access; it does not delete the user account or tenant-owned CRM records.

## Foundation

- Prisma owns the relational data model.
- App Router pages render the initial CRM views.
- REST endpoints live under `/api/v1/workspaces/:workspaceId`.
- `zod` validates incoming request payloads.
- `AuditLog` records workspace-scoped mutations.
- Authentication is represented by users, sessions, password reset tokens, and memberships. `AUTH_MODE=local` enables the built-in email/password login for existing users with password hashes and stores signed cookie sessions in the database. Expired sessions are rejected at request-context resolution and old expired rows are pruned during login. Password reset stores only hashed, expiring, one-time reset tokens; development/test can display reset links for manual QA without webhook config, while production never displays reset links and sends email only through the configured password-reset auth webhook with safe HTTPS reset URLs. `AUTH_MODE=trusted-header` requires an explicit safe `AUTH_USER_ID_HEADER` supplied by a trusted upstream/session layer. `AUTH_MODE=demo` uses the seeded actor and workspace fallback for local/demo work and emits a warning in production-like runtime.
- Workspace role helpers live in `lib/workspace-roles.ts` and currently cover role labels, member/admin/owner checks, a settings-management capability flag, and an ownership-transfer capability flag.
- Workspace role mutation is intentionally narrow: owners can promote/demote Admin/Member roles and transfer ownership; admins retain settings access but cannot promote/demote admins or transfer ownership.
- Workspace switching is selective: users can choose among existing memberships and workspaces they create or accept by invitation, but cannot edit roles or manage billing from the switcher.
- Runtime environment validation lives in `lib/env.ts`, and `/api/health` provides a non-sensitive readiness check for env validity plus database connectivity.
- CSV formatting lives in `lib/csv.ts`; workspace-scoped export assembly lives in `lib/services/export-service.ts` and is exposed through the workspace API boundary. Exports append deterministic custom field columns from the current workspace and record type only.

## Core Workflows

- Deals are the central sales workflow. The list uses normalized URL-driven state for search, filters, sort direction, custom field exact-value filters, and page size.
- Deal list-state constants live in `lib/deal-list-state.ts` so the Deals page, saved views, and reports share one set of supported filters and sorts.
- Products are workspace-scoped catalog records with required name, optional description, unit price, currency, and an active flag. Products can be edited and deactivated/reactivated; inactive products remain visible but cannot be added to new deal line items. Deal line items snapshot product name, price, and currency at creation so historical deal detail stays stable if the product changes later. Line item totals are displayed on deal detail only and intentionally do not overwrite `Deal.valueCents`, saved views, exports, or reporting totals. Quote drafts snapshot deal line items into `Quote` and `QuoteItem` records for internal review; quote detail, print, and PDF routes resolve through workspace-scoped quote/deal lookup and do not sync back to products or deal line items. Draft quotes can store one quote-level discount and one quote-level tax adjustment on `Quote`; percent values are stored as basis points, fixed values are stored as cents, calculated discount/tax cents are snapshotted, percent tax is calculated after discount, and `Quote.totalCents` remains the final stored total used by accepted-quote deal value sync. `QuotePublicLink` stores revocable secure random tokens for `/q/:token` quote pages; authenticated workspace members manage links through the workspace API, while public lookups return safe 404s for revoked, expired, missing, or malformed tokens. Public quote pages are standalone noindex views and intentionally omit the app shell, internal navigation, audit data, workspace member data, and rendered internal IDs. Public acceptance is limited to active links for `SENT` quotes, writes a `quote.public_accepted` audit event without an app user actor, and does not auto-sync the quote total to the deal. Quote status transitions are limited to `DRAFT -> SENT -> ACCEPTED` or `DRAFT -> SENT -> DECLINED`; accepted and declined quotes are terminal. Accepted quotes can be explicitly synced to the linked deal value and currency, which then affects reports and exports through the normal deal value field. Repeated syncs are no-ops when the deal already matches the quote. Quote print views use browser print styling, while quote PDF downloads are generated on demand by a small internal PDF writer with no file storage, browser automation, or email delivery.
- Activities, notes, and manual email logs attach to deals, contacts/people, organizations, and unconverted leads. Converted leads are read-only for new follow-up context; new activity, note, or email context should move to the converted deal.
- Manual email logs are workspace-scoped plain-text records with direction, occurred date, optional participant text, and CRM record attachments. They are created from record detail pages through the workspace API and intentionally do not send email, run whole-inbox sync, or run background capture. Email logs are snapshots; later template edits do not mutate already-created logs. Workspace-scoped email templates are reusable subject/body text snippets for manual logging only; they can be edited or deactivated from Settings, deactivated templates are not offered in record log forms, and templates do not include merge variables.
- Email connection metadata is represented separately from manual logs. `EmailConnection` tracks workspace-scoped provider readiness/status for Gmail / Google Workspace, Microsoft 365 / Outlook, and future IMAP / SMTP. `EmailConnectionSecret` stores OAuth access and refresh tokens only as AES-256-GCM encrypted payloads derived from `EMAIL_TOKEN_ENCRYPTION_KEY`; plaintext tokens, client secrets, and raw provider message payloads are not persisted. Gmail has a narrow OAuth connect/callback path using profile/email and Gmail metadata scopes plus a manual recent metadata sync action. That sync fetches a tiny recent Gmail batch, matches only known workspace contact emails, deduplicates by provider message id, and creates conservative `EmailLog` records with subject, participants, date, snippet, and provider ids. Microsoft and IMAP remain disabled/planned.
- Record timelines are read-only projections built from existing notes, activities, manual email logs, and audit events. They use deterministic timestamp ordering with stable tie-breaking, but they are not a separate event-sourcing layer.
- Custom fields are workspace-scoped definitions and values for Deals, Contacts/People, Organizations, and Leads. Unsupported field types remain read-only in the UI.
- Import/export is deliberately narrow: Deals, Contacts/People, Organizations, and Leads can be downloaded as deterministic CSV from Settings with record-type custom fields appended after core columns. Organizations have a pasted CSV workflow that validates `name`, accepts optional `domain`, reports unsupported columns, detects workspace-scoped duplicate names, and creates only rows still valid after server-side revalidation. Contacts follow the same pasted CSV pattern: validate names, detect email duplicates, check organization-name references, and create only rows still valid after server-side revalidation. Leads validate title/status, detect duplicate titles, check organization-name references, and create only valid non-duplicate leads after server-side revalidation. Deals validate title, explicit pipeline/stage references, optional value/status/date/owner/contact/organization references, skip invalid or duplicate rows, and create only rows still valid after server-side revalidation.

## Import/Export Rules

Organizations import came first because it has the smallest safe CRM record shape: a required name, optional domain, and no required parent record. Current import rules are intentionally conservative:

- Input is pasted CSV text only; file upload/storage and background jobs are not part of the current design.
- Organizations, Contacts, Leads, and Deals imports are create-only.
- Existing records are never overwritten, updated, or merged.
- Duplicate organization names, contact emails, lead titles, and deal title/pipeline/stage/contact/organization composites within the current workspace or pasted CSV are skipped with stable skip reasons.
- Invalid rows are skipped with stable validation reasons.
- Custom field columns are reported as unsupported and ignored.
- Imported records write workspace-scoped audit logs with consistent CSV import metadata: `organization.imported`, `contact.imported`, `lead.imported`, and `deal.imported`.

Contacts import accepts `name` or `firstName`, optional `lastName`, `email`, `phone`, and `organizationName`; duplicate emails are detected case-insensitively within the current workspace. Organization references are matched by normalized organization name in the current workspace only. Missing or ambiguous organization references are reported as invalid, and organizations are never auto-created from a Contacts import. Existing contacts are never overwritten, updated, or merged. Custom field import remains out of scope.

Leads import accepts `title` or `name`, optional `source`, `status`, and `organizationName`; duplicate titles are detected by normalized title within the current workspace because the Lead model has no direct email field. Blank status imports as `NEW`; accepted statuses are `NEW`, `QUALIFIED`, and `DISQUALIFIED`. Organization references are matched by normalized organization name in the current workspace only. Missing or ambiguous organization references are reported as invalid, organizations are never auto-created, and existing leads are never overwritten, updated, merged, or converted. `CONVERTED` status and conversion behavior are intentionally excluded from CSV import.

Deals import accepts `title` or `name`, required `pipeline` or `pipelineName`, required `stage` or `stageName`, and optional `status`, `value`, `currency`, `expectedCloseAt`, `contactEmail`, `contactName`, `organizationName`, and `ownerEmail`. Pipeline and stage must resolve inside the current workspace, and the stage must belong to the resolved pipeline. Owner, contact, and organization references must already exist in the current workspace; cross-workspace, missing, or ambiguous references are invalid. Contacts, organizations, and leads are never auto-created from Deals import. Deal custom field import remains deferred, and `Custom: ...` columns are reported as unsupported and ignored.

## Technical Debt

- Prisma CLI configuration lives in `prisma.config.ts`, including the seed command used by `npm run prisma:seed`.
- Deployment/readiness steps are documented in `docs/deployment-readiness.md`; hosting-provider-specific infrastructure is intentionally not included yet.

## Future Extensions

The schema leaves room for broader email sync, calendar sync, automations, webhooks, reporting, products, quotes, projects, campaigns, and marketplace integrations. Gmail sync should continue from the existing manual metadata action toward job-backed, reviewable, conservative matching rules before any full mailbox or provider expansion work. Redis and BullMQ should be introduced when background jobs become part of the milestone.
