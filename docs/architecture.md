# Architecture

This CRM starts as a single Next.js TypeScript application with API route handlers, Prisma, and PostgreSQL. Backend code is organized around workspace-scoped service modules so it can later move into a dedicated API service without changing the public REST contract.

## Tenancy

Every major CRM record belongs to a `Workspace`. Current request context is resolved in `lib/auth/request-context.ts`, which uses the narrow session abstraction in `lib/auth/session.ts`. Local mode resolves a signed httpOnly session cookie to a non-expired, non-revoked server-side `Session` row and active `User`; trusted-header values are ignored in local mode unless a valid local session cookie is also present. Trusted-header mode maps a configured upstream user-id header to an active `User`. Seeded demo fallback is available only in demo mode. Workspace-scoped page/API work must resolve an active `WorkspaceMembership`. The resulting `{ workspaceId, actorUserId }` actor is passed into service functions. Services are still responsible for applying workspace filters, verifying membership with `ensureWorkspaceAccess`, and excluding soft-deleted records.

Workspace-scoped API boundaries use consistent access semantics: missing sessions return `401`, authenticated non-members return `403`, and missing or cross-workspace records return non-leaky `404` responses where resource existence should not be exposed.

Browser workspace selection is stored in an httpOnly `northstar_workspace` cookie containing only the selected workspace id. The cookie does not grant access: `getCurrentWorkspaceContext` accepts it only when the current user has an active membership in that workspace. Invalid, stale, or non-member selections fall back to the configured demo/default workspace when valid, then to the user's first active membership. API routes remain explicitly workspace-id scoped and do not use the browser active-workspace cookie.

Workspace creation is intentionally narrow. A signed-in user can create a workspace from Settings with a required trimmed display name capped at 120 characters. The creator receives an `OWNER` membership, the workspace creation is audit-logged, and the new workspace id is written to the same verified active-workspace cookie used by switching. Duplicate workspace names are allowed; the service generates a unique slug by suffixing when needed. Creation does not invite members, edit roles, transfer ownership, or create seed CRM data.

Workspace invitations are also narrow. `WorkspaceInvitation` records are workspace-scoped, store normalized invitee email, target role, status, inviter, and timestamps, and currently support `PENDING`, `ACCEPTED`, and `REVOKED`. Owners/admins can create or revoke pending invitations from Settings. Invites do not grant `OWNER`; when auth email delivery is configured, creation queues a `workspace.invitation_email` job through the same Resend/webhook worker path as password reset email, and Settings still exposes a manual accept link for fallback sharing. Invited people without accounts can sign up with the invited email before accepting. A signed-in user can accept only an invitation whose email matches their account. Acceptance creates membership idempotently only while the accepted membership still exists, marks the invitation accepted, audit-logs the event, and selects the invited workspace through the same verified active-workspace cookie. If that member is later removed, the old accepted link cannot be reused to rejoin.

Workspace member management remains conservative. Owners/admins can remove normal members from Settings, owners can remove admins, and removal writes a `workspace_member.removed` audit log. Workspace owners can change another member between `ADMIN` and `MEMBER`, while normal role editing never assigns `OWNER`; those changes write `workspace_member.role_updated` audit logs. Ownership transfer is owner-only, targets another active membership, makes the target `OWNER`, and demotes the previous owner to `ADMIN` with a `workspace_member.ownership_transferred` audit log. The service blocks normal members from removing others, blocks admins from removing other admins, blocks non-owners from ownership transfer, keeps current owner removal blocked, and refuses changes that would leave the workspace without an owner/admin. Removing a member only removes access; it does not delete the user account or tenant-owned CRM records.

## Foundation

- Prisma owns the relational data model.
- App Router pages render the initial CRM views.
- REST endpoints live under `/api/v1/workspaces/:workspaceId`.
- `zod` validates incoming request payloads.
- `AuditLog` records workspace-scoped mutations.
- Authentication is represented by users, sessions, password reset tokens, and memberships. `AUTH_MODE=local` enables built-in email/password signup and login, creates a clean first workspace for new signups, and stores signed cookie sessions in the database. Expired sessions are rejected at request-context resolution and old expired rows are pruned during login. Password reset stores only hashed, expiring, one-time reset tokens; development/test can display reset links for manual QA without delivery config, while production never displays reset links and sends email only through the configured Resend or auth webhook delivery path with safe HTTPS reset URLs. `AUTH_MODE=trusted-header` requires an explicit safe `AUTH_USER_ID_HEADER` supplied by a trusted upstream/session layer. `AUTH_MODE=demo` uses the seeded actor and workspace fallback for local/demo work and emits a warning in production-like runtime.
- Workspace role helpers live in `lib/workspace-roles.ts` and currently cover role labels, member/admin/owner checks, a settings-management capability flag, and an ownership-transfer capability flag.
- Workspace role mutation is intentionally narrow: owners can promote/demote Admin/Member roles and transfer ownership; admins retain settings access but cannot promote/demote admins or transfer ownership.
- Workspace switching is selective: users can choose among existing memberships and workspaces they create or accept by invitation, but cannot edit roles or manage billing from the switcher.
- Runtime environment validation lives in `lib/env.ts`, and `/api/health` provides a non-sensitive readiness check for env validity plus database connectivity.
- CSV formatting lives in `lib/csv.ts`; workspace-scoped export assembly lives in `lib/services/export-service.ts` and is exposed through the workspace API boundary. Exports append deterministic custom field columns from the current workspace and record type only, quote delimiter/newline cells, and prefix spreadsheet formula-looking values with an apostrophe so user-entered CRM data opens as text in spreadsheet tools.
- The authenticated app shell keeps left navigation to major product destinations only: Dashboard, Assistant, Inbox, Pipeline/Deals/Quotes, Contacts, Organizations, Leads, Activities, Meeting Intelligence, and Settings. Search, setup/admin surfaces, provider cards, Smart Labels, AI Reply Assistant, Relationship Inbox, email follow-up drafting, and similar workflow-specific features stay inside their owning module pages or command/search surfaces instead of becoming separate primary nav items.

## Core Workflows

- Deals are the central sales workflow. The list uses normalized URL-driven state for search, filters, sort direction, one-field custom field filters, and page size.
- Deal list-state constants live in `lib/deal-list-state.ts` so the Deals page, saved views, and reports share one set of supported filters and sorts.
- List pages share query parsing in `lib/list-page-query.ts`. Filter-aware CSV exports use the same list-state option modules and service-filter conversion, with pagination stripped at the export link boundary so exports include all rows matching the current search/filter/sort.
- Products are workspace-scoped catalog records with required name, optional description, unit price, currency, and an active flag. Products can be edited, deactivated/reactivated, and exported; inactive products remain visible but cannot be added to new deal line items. Deal line items snapshot product name, price, and currency at creation so historical deal detail stays stable if the product changes later, and line item add/remove mutations are limited to open deals through the normal closed-deal edit lock. Product prices, line-item quantities, and computed line-item totals are rejected before they can overflow current integer storage. Line item totals are displayed on deal detail and exported as count/context columns only; they intentionally do not overwrite `Deal.valueCents` or reporting totals. Quote drafts snapshot deal line items into `Quote` and `QuoteItem` records for internal review; draft quote item edits from quote detail update only that quote snapshot, while sent, accepted, and declined quote items stay locked. Quote detail, print, and PDF routes resolve through workspace-scoped quote/deal lookup and do not sync back to products or deal line items. Draft quotes can store one quote-level discount and one quote-level tax adjustment on `Quote`; percent values are stored as basis points, fixed values are stored as cents, calculated discount/tax cents are snapshotted, percent tax is calculated after discount, and `Quote.totalCents` remains the final stored total used by accepted-quote deal value sync. Quote subtotals, fixed adjustment values, calculated discount/tax amounts, and totals are rejected before they can overflow current integer storage. `QuotePublicLink` stores revocable secure random tokens for `/q/:token` quote pages; authenticated workspace members can generate links only while a quote is `SENT` and can revoke active links through the workspace API, while public lookups return safe 404s for draft, revoked, expired, missing, or malformed tokens. Public quote pages are standalone noindex views and intentionally omit the app shell, internal navigation, audit data, workspace member data, and rendered internal IDs. Public acceptance is limited to active links for `SENT` quotes and writes a `quote.public_accepted` audit event without an app user actor. Quote status transitions are limited to `DRAFT -> SENT -> ACCEPTED` or `DRAFT -> SENT -> DECLINED`; accepted and declined quotes are terminal. Sent quotes capture the linked deal value/currency baseline; internal or public acceptance automatically syncs accepted quote total to the open deal when that baseline still matches, writes a `deal.value_synced_from_quote` audit event, and stores `dealValueSyncedAt` for idempotency. If the deal value changed after send, acceptance records `dealValueSyncConflict` and a `quote.deal_value_sync_conflict` audit event instead of silently overwriting; quote detail can then perform an explicit reviewed sync. Closed deals reject sync through the normal deal edit lock. Quote print views use browser print styling, while quote PDF downloads are generated on demand by a small internal PDF writer with no file storage, browser automation, or email delivery. Shared commercial readiness logic lives in `lib/commercial-workflow.ts` so deal detail, quote detail, Deals list filters, dashboard commercial metrics, and exports use the same v1 model for scoped line items, quote status, customer context, accepted-quote value alignment, and contract/SOW state.
- Contract workflow is deal-scoped and workspace-scoped through `DealContractStep`. Each open deal can have one `NDA`, `MSA`, and `SOW` step with status, owner, due/sent/signed dates, notes, and an external reference placeholder. The service layer enforces deal/workspace scope, closed-deal edit locking, owner membership, and the intended NDA -> MSA -> SOW sequence for forward-moving statuses. Audit events record contract-step creation, normal updates, and status changes. Deal lists and pipeline cards use batch-loaded contract-step summaries for lightweight next-step visibility. OpenContracts templates, document generation/storage, redlining, approvals, and signatures remain a future integration layer; see `docs/opencontracts-contract-workflow-integration.md` for the planned boundary.
- Optional vertical setup for supply-chain implementation consulting lives in `lib/services/supply-chain-vertical-service.ts`. It uses existing workspace-scoped custom fields, saved views, and product catalog records; it is idempotent, avoids schema changes, and reports unsupported/deferred categories instead of creating warehouse, project-management, ticketing, e-signature, or integration subsystems.
- Activities, notes, and manual email logs attach to open deals, contacts/people, organizations, and unconverted leads. Closed deals and converted leads are read-only for follow-up context: existing history stays visible, while new activity/note/email-log attachments, deal-attached activity changes/removal/completion, and note removal are blocked through normal APIs. New activity, note, or email context for a converted lead should move to the converted deal.
- Meeting Intelligence is a persisted review-first CRM intake workflow. `MeetingIntake` stores workspace, creator, source metadata, context, raw text, normalized markdown, status, proposal JSON, apply-result JSON, and failure messages. The domain layer under `lib/meeting-intelligence/*` handles source detection, local processors, provider boundaries, private provider-file storage, markdown normalization, deterministic CRM matching, provider-ready analysis output, and proposal generation. The service layer creates intakes, enforces workspace access, stores failed/provider-required processor states honestly, validates manually submitted apply targets, and applies only user-approved updates through existing note/activity/contact services so record locks, audit behavior, relationship profile validation, and attachment validation stay centralized. The v3 implementation processes pasted text, markdown, text files, RTF, HTML, CSV, JSON, text-based PDFs, and DOCX files synchronously with bounded local extraction; configured image/scanned-PDF/audio/provider-capable video artifacts are stored in the selected private Meeting Intelligence storage backend, local filesystem or S3/R2-compatible object storage, queue `meeting_intake.extract_media` jobs with a storage reference instead of large base64 payloads, and rejoin the same markdown/review pipeline after the worker retrieves and verifies the bytes. Northstar includes an internal provider-compatible OpenAI route for image/whiteboard extraction and audio transcription; scanned PDFs and video remain explicitly unsupported on that internal route until safe PDF OCR and video audio-extraction paths, or a provider capable of those source types, are added. Review proposals include evidence/confidence metadata for notes, activities, associations, and Relationship Brief updates, and can be edited, excluded, or reassigned before apply. Proposal generation distinguishes curated contact Relationship Brief updates from object-specific fact notes: contact/person facts can also be proposed as contact notes, organization/company facts become organization notes, deal-specific facts become deal notes, and lead-specific facts become lead notes when lifecycle rules allow them. Approved Meeting Intelligence meeting logs and next-step follow-ups are ordinary `Activity` rows created through the activity service; the completed meeting can also write `MeetingActivityAssociation` rows for cross-record timeline visibility, while next-step follow-ups keep source context in the activity description and remain completable through normal activity semantics. Relationship Brief retargeting uses the existing workspace-scoped people API to reload the selected contact's current brief for UI-only duplicate/staleness/merge previews, while final apply still revalidates and reloads the submitted contact server-side before merging. Approved Meeting Intelligence Relationship Brief field changes and manual contact-page Relationship Brief saves both reuse the existing `AuditLog` table by writing compact `person.updated` metadata with previous/new values, source context, actor, and timestamp; Meeting Intelligence entries also include accepted reviewed facts and intake context. Relationship Brief field semantics live in `lib/relationship-brief-usage.ts`, which defines deterministic labels, badges, customer-facing usage, AI/email usage, and internal-only/do-not-mention guidance for contact UI and downstream prompt construction. Apply results and contact detail render concise excerpts from audit metadata, with source/field filtering, changed-field summaries, expandable read-only source details for provenance, accepted facts, previous/new values, actor/timestamp, and source intake context, plus field-level usage guidance when available. Opt-in OpenAI semantic Relationship Brief extraction can enrich matched-contact proposals with sensitivity guidance, provider warnings, and merge previews, but deterministic proposals remain the fallback and provider output is validated before review. PPTX/XLSX conversion, missing media-provider config, missing/expired stored files, checksum mismatches, unsupported internal-route media paths, and provider failures are explicit boundaries and are not faked. File storage is temporary private extraction infrastructure, not user-facing document management; users review extracted markdown and editable proposals before any CRM mutation.
- Manual email logs are workspace-scoped plain-text records with direction, occurred date, optional participant text, provider metadata, and optional CRM record attachments. They are created from record detail pages through the workspace API and intentionally do not send email, run background capture, or mutate provider mail. Provider-synced logs can also store provider message/thread ids, provider snippets, and provider labels such as Gmail `UNREAD`; this supports the `/email` Full Inbox thread reader without adding a second inbox table. Email logs are snapshots; later template edits do not mutate already-created logs. Smart Label snapshots live inside existing `EmailLog.smartLabelJson` and are backward-compatible: the original flat `evidence` array remains the fallback, while newer snapshots can also include `categoryEvidence` plus validated `signalEvidence` entries keyed to selected allowed signals with short reasons and capped excerpts. Unknown or unselected signal evidence is discarded during normalization, and no exact text offsets are stored. Reviewed Relationship Inbox follow-up activities are associated back to their source email through `EmailLogActivityLink` provenance rows, while normal activities, including Meeting Intelligence-created follow-ups, remain usable without email provenance. Email follow-up history reads those links through the active workspace, opens activities through the existing activity edit/review route, can complete open linked activities through the existing workspace activity PATCH path, and treats legacy `Source email: ...` marker matches only as a fallback when no durable link exists. The Relationship Inbox priority service also computes deterministic queue explainers and next-best-action summaries from saved Smart Labels, saved category/signal evidence, flat fallback excerpts, CRM linkage, and durable follow-up state, including source/severity metadata, a compact evidence list, a full typed evidence trail with optional category/signal keys, saved excerpts, linked follow-up metadata, anchor targets for the stored email evidence section or linked records/activities, and an action-explanation model that maps the selected action to contributing signals, category, CRM link state, durable/legacy follow-up source, open/completed counts, and deterministic "Why this action?" reason copy. The UI displays the label/reason/evidence/action as review guidance only, without sending email or creating/updating CRM records unless the user explicitly submits the Gmail reply form or clicks the normal activity completion control. AI Email Reply Assistant prompt assembly consumes relevant Meeting Intelligence summaries, recent activities/follow-ups, and Relationship Brief facts through `relationshipBriefPromptFact`, including field-level usage language and withholding raw internal-only guidance from customer-facing draft context. Workspace-scoped email templates are reusable subject/body text snippets for manual logging only; they can be edited or deactivated from Settings, deactivated templates are not offered in record log forms, and templates do not include merge variables.
- The Northstar Assistant operating layer lives in `lib/services/northstar-ai-service.ts` and is documented in `docs/ai-operating-layer.md`. It is a reusable diagnostics and explanation foundation, not a separate mutation path. Context adapters assemble bounded, workspace-scoped, sanitized context for contact, deal, lead, organization, inbox/email, Meeting Intelligence proposal, and job/system surfaces. Deterministic detectors produce findings for missing CRM links, stale or overdue follow-ups, misplaced contact/company Relationship Memory signals, unlinked priority email, missing durable follow-ups, provider connection readiness, failed/dead/stale jobs, and Meeting Intelligence proposal warnings. Suggested actions are typed proposals with `reviewFirst: true`; applying them must go through existing service mutations so record locks, converted-lead restrictions, validation, and audit behavior remain centralized. Optional OpenAI summarization is provider-gated and receives only sanitized assistant context; it may rewrite explanatory text and cautions but cannot apply changes. The reusable panel currently appears on Inbox, Contact detail, and Deal detail pages and shows inspected context, findings, evidence, confidence, guardrails, and review-first next actions. The left-nav `/assistant` command console is backed by deterministic services under `lib/services/assistant/*`; it answers today's agenda, highest-risk open deals, and stored-email reply checks, and it can draft review-first action previews for activities, notes, contact Relationship Memory/profile guidance, AI preference changes, and supported contact/organization record improvements. Activity and note drafts use the durable `AssistantActionRequest` queue with explicit confirmation apply for eligible low-risk requests. Contact and organization create/update/link drafts route to reusable CRM Change Proposals instead of an Assistant-only mutation path. Those proposals store sanitized source/rationale/evidence, current and proposed values, permission state, duplicate/stale warnings, creator/workspace, and timestamps; apply/reject is explicit, audited, server-side permission checked, and applied creates link back to the final CRM record. Saving/rejecting requests or proposals does not call providers, sync/send email, save settings, mutate provider mail, or create/update/delete/link/convert/close unsupported CRM records.
- Email connection metadata is represented separately from manual logs. `EmailConnection` tracks workspace-scoped provider readiness/status for Gmail / Google Workspace, Microsoft 365 / Outlook, and future IMAP / SMTP. `EmailConnectionSecret` stores OAuth access and refresh tokens only as AES-256-GCM encrypted payloads derived from `EMAIL_TOKEN_ENCRYPTION_KEY`; plaintext tokens, client secrets, and raw OAuth tokens are not persisted. Disconnect deletes the encrypted provider secret and soft-deletes the connection metadata, while leaving provider-synced `EmailLog` snapshots available for workspace review; reconnecting the same account clears the soft-delete marker, writes fresh encrypted tokens through the existing OAuth upsert path, and queues a scoped Gmail sync job when Gmail is connected. Gmail uses the existing `/api/email-connections/google/connect` and `/api/email-connections/google/callback` routes with profile/email plus Gmail read/send scopes. The `/email` Inbox module separates Full Inbox, the synced Gmail thread reader, from Relationship Inbox, the classified CRM action queue; Smart Labels, AI Reply Assistant, and email follow-up creation/history are presented as sub-features inside that same Inbox page. The connected page renders a dense email-client shell with a compact header, toolbar, All-tab default thread list, URL-driven search/filter/sort, and page-based pagination instead of CRM-style cards for primary message rows. Empty sort state resolves to newest received first; priority and unread-first ordering are explicit user choices. The toolbar includes a URL-backed hide-unimportant filter driven by deterministic local metadata for newsletters, promotions, no-reply/automated blasts, receipt/status-style updates, and personal/non-work messages only when no CRM/action signal is present; this metadata never mutates Gmail labels, provider mail, or CRM records. It does not auto-select a first message; a `thread` query opens a focused full-page reader with a Back to inbox link that preserves selected account, tab, search, filters, sort, page, and page size. Pagination is applied over the stored synced thread set visible to the current authenticated user/workspace, so already-stored synced emails are reachable across pages; historical mailbox depth still depends on bounded sync/load-more rather than an unbounded provider crawl. It renders connection/readiness states from existing provider metadata: setup missing, token encryption missing, reconnect required, connected, sync issue, first sync needed, and no messages after sync. Gmail Full Inbox sync runs through the explicit `email.gmail_sync` job handler: the job payload contains only workspace/connection ids, token resolution stays inside the service layer, sync prefers Gmail history cursors stored as `historyId:<id>` in `EmailConnection.lastSyncCursor`, and expired/missing history falls back to a bounded recent `INBOX` batch. The default recent Gmail inbox bound is 75 messages; job-backed sync and load-more paths cap at 100 messages. The `/email` Sync Gmail inbox action only enqueues the selected Gmail connection, or all current-user Gmail connections, and redirects to the progress panel; provider calls run when `jobs:work` or `jobs:run-once` claims the durable job. The worker auto-enqueues due connected Gmail accounts on a conservative interval, uses the Gmail connection dedupe key to suppress concurrent duplicates, respects job retry/backoff/dead-letter semantics, and skips recently attempted terminal failures so revoked credentials cannot spin in an infinite enqueue loop. Sync attempts and completed results persist safe `EmailConnection` metrics: last attempted time, last successful sync time, imported count, duplicate count, skipped/unloaded message counts, fetched count, and sync mode; provider cards show those latest completed results separately from current job status. Provider-card sync status is looked up by the selected Gmail connection's dedupe key, not by the latest workspace Gmail job, so old jobs from another connection row cannot mask the clicked account's result. User-triggered load-more calls Gmail messages search with `INBOX` plus a bounded `before:YYYY/MM/DD` query derived from the oldest stored inbox message, while selected-thread refresh calls Gmail thread detail for a provider thread id that already has a same-workspace `EmailLog`; both paths reuse the same deduped `EmailLog` persistence, update connection health timestamps, and deliberately leave `lastSyncCursor` unchanged. Sync reads full message payloads when available, stores deduped `EmailLog` rows for matched and unmatched inbox messages, preserves provider labels/snippets/message ids/thread ids, and attaches same-workspace CRM links only when a known contact match is unambiguous. Explicit Gmail replies use the connected account and Gmail `messages.send`, require a user-submitted reply body, preserve the provider thread id when available, and log the sent message as an outbound `EmailLog`. Microsoft 365 / Outlook uses the same encrypted storage and safety model with Microsoft Graph, profile/mail read-only scopes, and the existing manual recent metadata sync action; it still stores only matched known-contact metadata/snippets and does not send replies. IMAP remains disabled/planned.
- Gmail Sync History and Health is a read-only product view derived from `EmailConnection` plus recent same-dedupe-key `Job` rows. It exposes only safe connection/job refs, status, manual/automatic/legacy source, latest counts, retry and next-auto-sync timing, active duplicate-job suppression, stale-worker hints, sanitized failure categories, and reconnect/retry guidance; it does not expose tokens, raw provider payloads, message bodies, stack traces, or full job payloads.
- Record timelines are read-only projections built from existing notes, activities, manual email logs, and audit events. They use deterministic timestamp ordering with stable tie-breaking, but they are not a separate event-sourcing layer.
- Custom fields are workspace-scoped definitions and values for Deals, Contacts/People, Organizations, and Leads. Deal custom field value updates honor closed-deal edit locking, converted leads reject value updates, and unsupported field types remain read-only in the UI.
- Import/export is deliberately narrow: Deals, Contacts/People, Organizations, Leads, Activities, Products, and Quotes can be downloaded as deterministic CSV from Settings. Deal, Contact/Person, Organization, and Lead exports append record-type custom fields after core columns, while Product and Quote exports are full workspace-scoped snapshots for those resources. Organizations have a pasted CSV workflow that validates `name`, accepts optional `domain`, reports unsupported columns, detects workspace-scoped duplicate names, and creates only rows still valid after server-side revalidation. Contacts follow the same pasted CSV pattern: validate names, detect email duplicates, check organization-name references, and create only rows still valid after server-side revalidation. Leads validate title/status, detect duplicate titles, check organization-name references, and create only valid non-duplicate leads after server-side revalidation. Deals validate title, explicit pipeline/stage references, optional value/status/date/owner/contact/organization references, skip invalid or duplicate rows, and create only rows still valid after server-side revalidation. CSV import creation is intentionally limited to Deals, Organizations, Contacts/People, and Leads; Activities, Notes, Products, Quotes, saved views, users, memberships, and custom-field imports are deferred.

## Import/Export Rules

Organizations import came first because it has the smallest safe CRM record shape: a required name, optional domain, and no required parent record. Current import rules are intentionally conservative:

- Input is pasted CSV text only; file upload/storage and background jobs are not part of the current design.
- Organizations, Contacts, Leads, and Deals imports are create-only.
- Existing records are never overwritten, updated, or merged.
- Duplicate organization names, contact emails, lead titles, and deal title/pipeline/stage/contact/organization composites within the current workspace or pasted CSV are skipped with stable skip reasons.
- Invalid rows are skipped with stable validation reasons.
- Custom field columns are reported as unsupported and ignored.
- Imported records write workspace-scoped audit logs with consistent CSV import metadata: `organization.imported`, `contact.imported`, `lead.imported`, and `deal.imported`.
- Supported camelCase CSV relationship/date headers also accept human-readable spaced aliases for spreadsheet-pasted imports, including owner, contact, organization, pipeline, stage, and expected close date columns.

Contacts import accepts `name` or `firstName`, optional `lastName`, `email`, `phone`, `organizationName`, and `ownerEmail`; duplicate emails are detected case-insensitively within the current workspace. Organization and owner references are matched in the current workspace only. Missing or ambiguous organization references are reported as invalid, and organizations are never auto-created from a Contacts import. Existing contacts are never overwritten, updated, or merged. Custom field import remains out of scope.

Leads import accepts `title` or `name`, optional `source`, `status`, `contactEmail`, `contactName`, `organizationName`, and `ownerEmail`; duplicate titles are detected by normalized title within the current workspace because the Lead model has no direct email field. Blank status imports as `NEW`; accepted statuses are `NEW`, `QUALIFIED`, and `DISQUALIFIED`. Contact, organization, and owner references are matched in the current workspace only. Missing or ambiguous references are reported as invalid, organizations are never auto-created, and existing leads are never overwritten, updated, merged, or converted. `CONVERTED` status and conversion behavior are intentionally excluded from CSV import.

Deals import accepts `title` or `name`, required `pipeline` or `pipelineName`, required `stage` or `stageName`, and optional `status`, `value`, `currency`, `expectedCloseAt`, `contactEmail`, `contactName`, `organizationName`, and `ownerEmail`. Deal values must fit the current integer-cent storage limit. Pipeline and stage must resolve inside the current workspace, and the stage must belong to the resolved pipeline. Owner, contact, and organization references must already exist in the current workspace; cross-workspace, missing, or ambiguous references are invalid. Contacts, organizations, and leads are never auto-created from Deals import. Deal custom field import remains deferred, and `Custom: ...` columns are reported as unsupported and ignored.

Meeting Intelligence intentionally reuses CRM import/export safety principles without becoming a generic import system. It is review/apply only, never updates records silently, never auto-creates contacts or organizations, and records unmatched or ambiguous entities as warnings for user correction.

## Technical Debt

- Prisma CLI configuration lives in `prisma.config.ts`, including the seed command used by `npm run prisma:seed`.
- Deployment/readiness steps are documented in `docs/deployment-readiness.md`; a minimal Railway preview configuration is included, while other hosting-provider infrastructure is intentionally not included yet.

## Future Extensions

The schema leaves room for broader email sync, calendar sync, automations, webhooks, reporting, products, quotes, projects, campaigns, and marketplace integrations. Gmail Full Inbox v1 now has job-backed incremental history sync with recent-batch fallback plus explicit cursor-safe older-message loading and selected-thread refresh; future email work should add persisted provider pagination tokens, provider read-state actions, attachment handling, and Microsoft parity only as explicit bounded slices. Redis and BullMQ should be introduced only if the database-backed worker becomes insufficient for the milestone.
